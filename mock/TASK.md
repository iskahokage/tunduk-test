# Тестовое задание для позиции React Middle

## Описание проекта

Необходимо реализовать модуль **Candidate Dashboard** — дашборд для HR-специалиста, который позволяет просматривать список кандидатов, анализировать их соответствие вакансии и управлять статусами обработки резюме.

Задание демонстрирует типичные задачи, с которыми сталкивается React-разработчик в команде CV-Scan.

---

## Технические требования

### Стек
- **React 18** + **TypeScript**
- **Vite** (сборка)
- **Redux Toolkit** или **Zustand** (state management)
- **React Router** (маршрутизация)
- **Tailwind CSS** или **SCSS** (стилизация)
- **Jest + React Testing Library** (тестирование)

### API
Для работы с данными используйте моки.

- **Основной мок** — `mock/candidates.json` (25 кандидатов, покрывает задания 1–4)
- **Большой мок** — `mock/candidates-large.json` (120 кандидатов, для задания 5 — оптимизация)

## Задания

### Задание 1. Список кандидатов (Основная страница)

**Требования:**
- Отобразить карточки кандидатов в виде таблицы или карточек (cards layout)
- Для каждого кандидата показать: ФИО, город, общий опыт, вердикт (цветовая индикация), основной стек
- Реализовать **фильтрацию по вердикту** (Все / ПОДХОДИТ / ЧАСТИЧНО / НЕ ПОДХОДИТ)
- Реализовать **поиск по ФИО** (debounce 300ms)
- Реализовать **сортировку** по имени, опыту, дате добавления
- Пагинация (10 кандидатов на странице) — в моке 25 кандидатов, 3 страницы

**Критерии оценки:**
- [ ] Корректная работа фильтрации и поиска
- [ ] Debounce реализован без лишних ререндеров
- [ ] Состояние фильтров сохраняется в URL (query params)
- [ ] Адаптивная вёрстка (мобильная + десктоп версия)
- [ ] Обработка пустого результата

---

### Задание 2. Карточка кандидата (Детальная страница)

**Требования:**
- При клике на кандидата открывается детальная страница `/candidate/:id`
- Отобразить полную информацию: контакты, опыт работы (список), образование, стек
- Блок **«Критерии оценки»** с цветовой индикацией (ok/partial/no)
- Блок **«Вопросы для собеседования»** — список вопросов
- Блок **«Summary»** — текстовое резюме
- Кнопка «Назад к списку» (с сохранением фильтров)

**Критерии оценки:**
- [ ] Корректная работа роутинга
- [ ] Обработка 404 (кандидат не найден)
- [ ] Семантическая вёрстка
- [ ] Разделение на компоненты (не монолитный файл)
- [ ] Корректное отображение цветовой индикации verdict/criteria

---

### Задание 3. Управление статусами

**Требования:**
- В детальной карточке кандидата добавить выпадающий список для смены статуса:
  - `Новый` → `На рассмотрении` → `Приглашён` → `Отклонён`
- При смене статуса отправить mock API запрос (имитация `PATCH /api/candidates/:id/status`)
- Отобразить статус в списке кандидатов (бейдж/иконка)
- Добавить оптимистичное обновление UI с откатом при ошибке

**Критерии оценки:**
- [ ] Оптимистичное обновление реализовано корректно
- [ ] Обработка ошибок (rollback состояния)
- [ ] Loading state во время запроса
- [ ] Уведомления об успехе/ошибке (toast/snackbar)

---

### Задание 4. Тестирование

**Требования:**
- Написать unit-тесты для компонентов:
  - Фильтрация кандидатов (по вердикту)
  - Поиск с debounce
  - Отображение цветовой индикации verdict
- Написать integration-тест для перехода список → детальная страница

**Критерии оценки:**
- [ ] Покрытие тестами ключевых функций
- [ ] Использование React Testing Library (не Enzyme)
- [ ] Моки API вызовов
- [ ] Проверка edge cases (пустой список, ошибка загрузки)

---

### Задание 5. Оптимизация производительности (дополнительно)

**Требования:**
- Переключиться на `mock/candidates-large.json` (120 кандидатов)
- Реализовать виртуализацию списка (react-window или аналог) — без неё список из 120 карточек тормозит
- Мемоизировать вычисления фильтрации/сортировки (useMemo)
- Избежать лишних ререндеров при смене фильтров (React.memo)
- Страница пагинации должна работать и с большим списком (12 страниц по 10)

**Критерии оценки:**
- [ ] Виртуализация работает корректно
- [ ] useMemo использован уместно (не чрезмерно)
- [ ] React.memo применён к тяжёлым компонентам
- [ ] Нет утечек памяти (cleanup в useEffect)

## Что оценивается

### Обязательно
- Чистота и читаемость кода
- Структура проекта и разделение ответственности
- Типизация TypeScript (никаких `any`)
- Работа с состоянием (без prop drilling)
- Обработка ошибок и loading states
- Адаптивная вёрстка

### Будет плюсом
- Использование custom hooks
- Оптимизация производительности
- Покрытие тестами >70%
- Доступность (a11y)
- Комментарии к сложным участкам кода
- Использование React Query для работы с API

---

## Критерии оценки

| Критерий | Junior | Middle | Strong Middle |
|----------|--------|--------|---------------|
| **Код** | Работает, но есть дубли, prop drilling | Чистый, переиспользуемые компоненты | Архитектурно грамотный, SOLID |
| **TypeScript** | Есть `any`, типы не везде | Полная типизация | Generics, utility types |
| **State Management** | useState везде, prop drilling | Redux/Zustand, нормализация | Оптимизация, селекторы |
| **Тесты** | Нет или 1-2 теста | Покрытие основных функций | Покрытие >70%, edge cases |
| **UI/UX** | Базовая вёрстка | Адаптивность, loading states | Анимации, a11y, polish |
| **Производительность** | Не задумывается | useMemo/useCallback | Виртуализация, профилирование |
